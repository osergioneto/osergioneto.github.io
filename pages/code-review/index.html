Introdução

O padrão do Code Review

Mentorando

O code review pode ter uma função importante para ensinar algo novo aos desenvolvedores, seja uma linguagem, um framework ou conceitos de engenharia de software.

Principios

  • Os fatos técnicos e os dados anulam as opiniões e preferências pessoais.
  • Estilo de código
  • Aspectos de design de software quase nunca são uma questão puramente de estilo ou apenas uma preferência pessoal.
  • Seja consistente

Rsolvendo conflitos

  • CONSENSO
    • Não deixe um PR ficar parado porque o autor e o revisor não concordam.

Design

  • As interações de várias partes do código no CL fazem sentido?
  • Essa alteração pertence à sua base de código ou a uma biblioteca?
  • Ele se integra bem com o resto do seu sistema?
  • Ele se integra bem com o resto do seu sistema?

Funcionalidade

  • Este Pull Request faz o que o desenvolvedor pretendia?
  • O que o desenvolvedor pretendia é bom para os usuários deste código?

Complexidade

  • O código é mais complexo do que deveria?
  • As linhas são muito complexas?
  • As funções são muito complexas?
  • As classes são muito complexas?

Testes

  • Os testes realmente falharão quando o código for quebrado?
  • Se o código mudar abaixo deles, eles começarão a produzir falsos positivos?
  • Cada teste fazem validações simples e úteis?
  • Os testes são separados adequadamente entre diferentes funções de teste?

Nomeação

  • O desenvolvedor escolheu bons nomes para tudo?

Nomeação

  • O desenvolvedor escreveu comentários claros em inglês compreensível?
  • Todos os comentários são realmente necessários?

Documentação

  • Atualizar/excluir documentação
  • Se faltar documentação, peça.

Todas linhas

  • Olhe cada linha do PR
  • Não ignore código feito por humano
  • Não presuma qualidade

Características de um bom PR

  • Elogie quando encontrar algo bom
  • O código tem o design bem feito
  • A funcionalidade é boa para os usuários do código
  • Todas as mudanças na UI são perceptíveis e bonitas
  • Qualquer concorrência é feita de maneira segura
  • O código não é mais complexo do que precisa ser
  • O desenvolvedor não está implementando coisas que ele pode precisar no futuro mas não tem certeza
  • O código tem testes apropriados

Características de um bom PR

  • O desenvolvedor usa nomes apropriados para tudo
  • Comentários são simples e úteis. Geralmente explicam o "porque" e não o "o que"
  • O código está documentado
  • O código está conforme os guias de estilo
